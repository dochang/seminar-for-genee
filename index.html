REST

RPC 的缺点

  • 客户端必须知道方法名
  • 方法参数的顺序、类型、个数有规定
  • 代码高度耦合
  • 不能缓存
  • 依赖多

REST 的优点

  • 简单易懂
  • 代码耦合度小
  • 支持缓存
  • 充分地利用 HTTP 协议，标准化的接口
  • 依赖少，客户端只需要解析 HTTP 消息

特征

  • 每个 URI 代表一个资源
  • 对资源的操作包括增删查改(CRUD)
  • 通过修改资源的表现形式来操作资源

要求

  • Client-Server
  • 无状态
  • 可缓存的
  • 统一接口
  • 分层系统
  • 可编程（可选）

统一接口

  • 资源的标识
  • 资源的修改
  • 足够的描述信息
  • 所有可用的操作都应该通过单一的入口点发现

要素

  • URI
  • Media Type
  • HTTP methods

示例

  • http://example.com/users
  • http://example.com/users/user17

http://example.com/users

GET 列出集合中的成员
POST 在集合中增加一个新的成员；自动分配新成员的 URI
PUT 替换整个集合
DELETE 删除整个集合

http://example.com/users/user17

GET 获取该成员的某种格式的数据
POST 通常用于在该成员下再建立子成员时使用
PUT 替换这个成员，如果不存在，则建立
DELETE 删除该成员

安全的方法

GET

幂等 (idempotent) 的方法

PUT, DELETE

POST

重要的状态码

  • 200 OK
  • 201 Created
  • 204 No Content
  • 303 See Other
  • 304 Not Modified
  • 400 Bad Request
  • 401 Unauthorized
  • 403 Forbidden
  • 404 Not Found
  • 405 Method Not Allowed
  • 409 Conflict
  • 415 Unsupported Media Type
  • 500 Internal Server Error

      "productId": 12343,

  ]

HTTP/1.1 201 Created
Location: /api/carts/323392

如何设计 REST API

  • 转化为资源（设计 URI）
  • 实现需要响应的请求方法 (GET/POST/PUT/DELETE)
  • 决定需要的格式 (html/json/xml/...)
